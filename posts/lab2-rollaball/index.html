Lab 2: Roll A Ball

The first practicle asignment of the class is to create a first Unity game.

Roll a ball the is a unity tutorial created for beginners to make their first game. The principle is simple: you have to create a plan with a ball and collectibles. In order to win the game the ball has to collect a certain number of collectible.

Creation of the project and first visual aspects

After creating the project, we need to add the first essential aspects of the game: the plane and the ball. The tutorial is very self explanatory and allows you to add a little bit of color/ creativity to your project.

The tutorial also helps you setting up the first script in order to control both the camera or the player.

Creating collectibles

Then comes the creation of the collectibles. That is when I decided to make it a little more spicy. In the game intended by unity, all the collectibles allow you to earn points to win.

I wondered, what is the point to play a game when you are sure of winning. I chose then to create two types of collectibles: ones that earns you point, painting in green (easy) and one that makes you lose points in red (duh).

The ball simply has to go over those collectibles to earn or lose points.

Generating collectibles

Now that the prefabs of the collectibles are created the idea is to place them on the plane. In the tutorial we can see them placing them in a circle. Once again this game can be pretty boring if the collectibles are always placed at the same place.

I thought of creating a script applied on the plane that in the creating of the plane will add randomly placed collectibles.

Here I had several choices:

  1. Ask for a overall number of collectibles and have it randomly mixed in negative and positive collectibles: This one might have been a little too extreme since it can lead to non winable games.
  2. Ask for a overall number of collectibles and have a pre-set percentage of negative collectibles: This possibility is actually really good since you can then decide the different percentages for different levels of playing for example.
  3. Ask for a number of negative collectibles and a number of negative collectibles: This is not the best one but this is the one I chose to have full control over each number of collectible.
for(int i=0 ; i< numberOfPositiveCollectibles ; i++){
    Instantiate(positiveCollectiblePrefab, randomPosition(), Quaternion.identity);
}
for(int i=0 ; i< numberOfNegativeCollectibles ; i++){
    Instantiate(negativeCollectiblePrefab, randomPosition(), Quaternion.identity);        }

Now, your collectibles change place at each start of the game and you can change the number of collectibles in the plane inspector.

Extending to terrain

After playing a few times with my new game, I though the space was too small and moreover the collectibles were a little to simple to find.

I though of using a terrain instead of a plane. This will allow me not only to make it bigger but also to add some height variation. Some hills, some slopes so that the ball can jump or else and be actually harder to control (for example down a slope the speed changes and is not only the one you input with the keyboard)

After creating my terrain, I have to color it otherwise it will get pretty doll. I got textures from the Unity store in order to find more realistic texture than just a simple color to add some dimension.

With this change we had to change the apparition of the collectibles since they were generated at 0 height which will not adapt with the new created hills/valleys.

First the size of the terrain was not the same as the plane which was just 10x10. Since we want the script to work on whatever size of the terrain:

x = (int)Random.Range(0,terrain_data.size.x);
z = (int)Random.Range(0,terrain_data.size.z);

Then we easily could get the height of the terrain at those particular x and z: (be careful that it is z,x and not x,z otherwise you will get the wrong height)

y = (int)terrain_data.GetHeight(z,x);

However, with this placement we got so weird thing happening with the collectibles and some were floating in the air as so:

This problem is due to the fact that the height is in the terrain space perpective and you have to transform to the world space perspective:

y = (int)terrain.SampleHeight(new Vector3(x,y,z));

We have now our collectible rightly placed all over the terrain at the right height.

Creating new camera control

We then face another problem caused by the fact that we made the terrain bigger. We cannot raise the camera so that we see the whole terrain and the ball. Doing so we will not see the ball at all. The issue we face due to the camera following the ball from behind is pretty simple:

Indeed, when we reach the end of the terrain or if we want to go backwards, we do not see anything and we can very easily hit a negative collectible.

The idea is to be able to move the camera and to be able to apprehend the whole terrain better and not only from one point of view.

The first thing we can do is being able to move the camera rotation to see better the sides.

We can use for example the mouse to move the camera:

float mx = Input.GetAxisRaw("Mouse X");
if (Input.GetMouseButton(1)) {
   rot_x -= my * look_sensitivity;
   rot_y += mx * look_sensitivity;
}

This will get us the movement of the mouse on the game screen. The Input.GetMouseButton(1) will check if the right button is clicked. Indeed we do not want the camera to move whenever you move your mouse. In order to control the rotation of the camera you right click and move the mouse.

This is a good start to apprehend better the environment and get a better sense of what is around. However it does not solve the problem we had for going backwards. Indeed, the camera will rotate but not change its position compared to the ball, so by rotating by 180 degres you will be in front of the ball and have no visual of it.

We need to add another control to move the position of the camera around the ball. The idea is to move in a circle around the ball. I am transforming the movement of the mouse into a rotation angle. With this angle we compute the new vector between the ball and the camera. We then just have to ajust the rotation to be able to see the ball. This movement is made when the left click is used. Here is the written code:

if(Input.GetMouseButton(0)){

   //Transforming the mouse movement to a rotation angle
   float rotationAngle = 3.14f/180.0f * mx * look_sensitivity;
   float[,] rotationMatrix = new float[2, 2];
   float angleToPlayer;

   rotationMatrix[0,0] = Mathf.Cos(rotationAngle);
   rotationMatrix[0,1] = - Mathf.Sin(rotationAngle);
   rotationMatrix[1,0] = Mathf.Sin(rotationAngle);
   rotationMatrix[1,1] = Mathf.Cos(rotationAngle);

   //Compute the new vector between the ball and the camera with the rotation matrix
   Vector2 newOffset = new Vector2(rotationMatrix[0,0]*offset.x + rotationMatrix[0,1]*offset.z ,
                        rotationMatrix[1,0]* offset.x + rotationMatrix[1,1]* offset.z);

   offset = new Vector3(newOffset.x, offset.y, newOffset.y);

   //Computer the rotation of the camera to see the ball
   if (offset.z <0  && offset.x < 0) angleToPlayer = Mathf.Atan(offset.x/offset.z)*180.0f/3.14f;
   else if (offset.z <0  && offset.x > 0) angleToPlayer = -  Mathf.Atan(Mathf.Abs(offset.x/offset.z))*180.0f/3.14f;
   else if (offset.z >0  && offset.x > 0) angleToPlayer = 180.0f + Mathf.Atan(Mathf.Abs(offset.x/offset.z))*180.0f/3.14f;
   else angleToPlayer =  180.0f - Mathf.Atan(Mathf.Abs(offset.x/offset.z))*180.0f/3.14f;

   //Update the rotation of the camera
   t.localRotation = Quaternion.Euler(rot_x, angleToPlayer, 0.0f);
}
//Update the position of the camera
t.position = player.transform.position + offset;

With those two camera movements, the user is now able to control freely and aprehend better the environment.

The terrain can now be modified to fit the game creator’s will. The game is now finished and ready to be played!
